# 跑得快记分结算 H5 — 设计文档

日期：2026-07-22
状态：已与用户确认

## 1. 背景与目标

用户和朋友线下玩"跑得快"扑克牌（每人 10 张，先出完者赢），每局结束需要结算：
所有未出完牌的玩家按剩余牌数把分记给赢家。以前用微信小程序或草稿纸记分，
但部分朋友用不了微信，草稿纸又麻烦。

目标：做一个**单手机记分**的 H5 应用，牌局结束后由一人录入各玩家剩余牌数，
程序自动完成每局结算、累计积分、以及整场结束后的净额转账方案。
**零服务器**：纯静态页面，数据存浏览器本地。

## 2. 游戏与结算规则

- 每人发 10 张牌，先出完的人赢，该局结束。
- 结算：赢家 0 张；每个输家按**剩余牌数 × 单价**付给赢家。
  例：张三赢，李四剩 4 张、王五剩 2 张、戴六剩 7 张 → 张三收 13 张牌的钱。
- **全关翻倍**：输家剩满 10 张（一张没出）按 20 张结算。
  剩 10 张时程序自动标记全关，录入者可手动取消（防特殊情况）。
- 单价在开场时设置（元/张，支持两位小数，如 1 元、0.5 元），一场内固定。
- 人数不定：支持 2～8 人。
- 每局保存完整转账明细（谁 → 谁、几张牌、多少钱），随时可查。

## 3. 使用模式

单手机记分员模式：一台手机打开页面，一人录入。多手机联机同步**不在本期范围**
（未来如需要，可在电脑上跑一个小服务端，本设计的数据结构不阻碍该扩展）。

## 4. 页面与交互流程（移动端竖屏优先）

### 4.1 首页
- 「开新一场」：勾选玩家（从历史名单选 + 可新增名字）→ 设置单价 → 开始。
- 「继续本场」：存在未结束的场时显示，一键回到记分主页。
- 「历史记录」：按日期列出已结束的场，可查看每场的局明细与最终转账方案。
- 「数据备份」入口：导出 / 导入 JSON。

### 4.2 记分主页（一场进行中）
- 实时积分榜：每人当前累计净输赢（牌数 + 金额），按输赢排序。
- 大按钮「记一局」。
- 局记录列表：每局一行（第 N 局 · 赢家 · 各输家牌数），点开看转账明细，
  支持**修改**与**删除**任意一局。
- 「玩家管理」：中途加人（下一局起生效）、标记离场（新局不再出现，
  历史与净额保留，最终结算仍包含）。
- 「结束本场」按钮。

### 4.3 记一局（目标 10 秒内完成）
1. 点选赢家（大头像/名字按钮）。
2. 其余在场玩家逐个输入剩余牌数：0～10 大数字快捷键盘；
   选 10 自动亮"全关 ×2"标记（可点击取消）；选 0 表示该玩家本局也出完/不计。
3. 预览本局转账明细（李四 → 张三 4 张 ¥4 …）。
4. 确认保存，返回记分主页。

### 4.4 结束本场
- 每人最终净输赢（牌数与金额）。
- **最简转账方案**：净额互抵后给出"谁转给谁多少钱"，笔数最少（≤ 人数-1 笔）。
- 「复制战绩」：生成纯文本（每人盈亏 + 转账方案），复制后发到任意聊天工具。
- 结束后该场进入历史，只读。

## 5. 数据模型（localStorage）

存储 key：`runfast.v1`。所有累计值**不落库**，一律从 rounds 实时重算，
保证修改/删除局后总分永远一致。

```json
{
  "version": 1,
  "playerDirectory": ["张三", "李四", "王五", "戴六"],
  "sessions": [
    {
      "id": "s1721800000000",
      "createdAt": "2026-07-22T20:00:00+08:00",
      "pricePerCardFen": 100,
      "players": ["张三", "李四", "王五", "戴六"],
      "activePlayers": ["张三", "李四", "王五", "戴六"],
      "status": "active",
      "rounds": [
        {
          "id": "r1",
          "at": "2026-07-22T20:10:00+08:00",
          "winner": "张三",
          "losers": [
            { "name": "李四", "cardsLeft": 4,  "shutout": false },
            { "name": "王五", "cardsLeft": 2,  "shutout": false },
            { "name": "戴六", "cardsLeft": 10, "shutout": true  }
          ]
        }
      ]
    }
  ]
}
```

- 金额一律以**分**（整数）运算，避免浮点误差；展示时格式化为元。
- `losers[].shutout` 为最终生效值（默认 cardsLeft===10 时 true，用户可改）。
- 玩家以名字为标识（同场内名字唯一，开场时校验）。
- 同一时间最多一个 `status: "active"` 的场。

## 6. 结算算法（纯函数，可测试）

```
countedCards(loser)  = loser.shutout ? 20 : loser.cardsLeft
roundTransfers(round, priceFen)
  = losers.map(l => { from: l.name, to: round.winner,
                      cards: countedCards(l), fen: countedCards(l) * priceFen })
sessionNet(session)  = 按人累加所有局的收支 → Map<name, fen>（总和恒为 0）
settleUp(netMap)     = 贪心：每次取最大欠款人与最大债权人配对，
                       转 min(欠, 债)，直至清零 → [{from, to, fen}]
```

不变量（测试断言）：每局转账总额等于赢家所得；全场净额之和为 0；
settleUp 结果按人汇总后与净额完全一致，且笔数 ≤ 人数-1。

## 7. 容错与边界

- 牌数输入仅允许 0～10 整数；赢家固定 0 张。
- 任意一局可改可删，总分从头重算。
- 中途加人：加入后出现在新局的录入名单；此前的局不受影响。
- 离场：不再出现在新局录入名单，净额保留并参与最终结算。
- 误触保护：删除局、结束本场需二次确认。
- localStorage 写入失败（隐私模式等）时明确提示用户。
- 数据可导出为 JSON 文件、可导入恢复（换手机/清缓存兜底）。

## 8. 技术方案

- **原生 HTML/CSS/JS，最终交付为单个 `index.html`**，无框架、无运行时依赖。
- 源码结构（开发期分文件，交付时内联合并）：
  - `src/logic.js` — 结算纯函数（浏览器/Node 双兼容，供测试直接 import）
  - `src/app.js` — UI 与状态管理
  - `src/style.css` — 样式（移动端优先，大按钮大字号）
  - `src/index.html` — 页面骨架
  - `build.js` — 约 20 行 Node 脚本，把 js/css 内联进 `dist/index.html`
- 测试：`node --test` 跑 `test/logic.test.js`，覆盖第 6 节全部不变量与
  全关、修改重算、settleUp 边界（0 元、单人赢家通吃、并列欠款）。

## 9. 交付方式（两者都要）

1. **公网链接**：`dist/index.html` 发布到飞书妙搭，生成公网 URL，
   任何手机浏览器打开即用，不依赖微信。
2. **单文件**：同一个 `dist/index.html` 直接发给朋友，本地打开完全离线可用。

注意：数据跟随"该手机的该浏览器"存储，清浏览器数据会丢，靠导出功能兜底。
（离线缓存 PWA、多手机联机同步为未来可选项，不在本期。）
